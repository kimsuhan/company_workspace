# 실행 환경

- 개발 서버는 작업자가 직접 실행하고, 작업 종료 전 브라우저에서 실제 화면을 확인한다.
- 개발 서버 실행 시 필요한 포트가 이미 사용 중이면 기존 서버를 종료한 뒤 다시 실행한다.
- 백엔드/프론트 포트를 변경하면 `apps/frontend/.env`의 `NEXT_PUBLIC_BACKEND_URL`이 실제 백엔드 `PORT`와 일치하는지 확인한다.
- SSE를 추가할 때는 idle 연결 끊김 방지를 위해 `retry`와 heartbeat를 함께 둔다.
- Project Health 동작/API/등록 방법은 `docs/project-health.md`를 기준으로 확인한다.
- 공통 파일 업로드 구조는 `docs/files/common-file-upload.md`를 기준으로 확인한다.
- PM2 배포 패키징/갱신 절차는 `docs/deploy/pm2.md`를 기준으로 확인한다.
- 반복되면 스킬화할 후보는 `docs/evolve/skill-candidates.md`에 누적하고, evolve 때 확인한다.

# 디자인 작업 지침

- UI, 스타일, 레이아웃, 색상, 타이포그래피, 컴포넌트 시각 규칙을 변경할 때는 `DESIGN.md`를 반드시 기준으로 따른다.
- 구현 과정에서 디자인 결이 `DESIGN.md`와 달라져야 한다면 코드만 바꾸지 말고 `DESIGN.md`도 함께 수정해 최신 디자인 기준을 반영한다.
- `DESIGN.md`와 코드가 충돌하면 먼저 `DESIGN.md`를 우선하고, 충돌이 의도된 변경이면 문서와 코드를 같은 작업 범위에서 맞춘다.

# 프론트 작업 방식

- 프론트는 먼저 실제 데이터 연동 없이 화면 껍데기를 구성한다.
- 사용자가 명시적으로 요청하기 전까지 API 연동, 인증, 상태관리, 라우팅 확장은 하지 않는다.
- 대시보드 첫 화면은 데스크톱 최대 화면에서 핵심 섹션이 한눈에 보이도록 구성한다.
- 메인 대시보드는 기본적으로 2x2 섹션 그리드를 우선 사용한다.
- 실제 데이터가 필요한 영역은 더미 데이터로 UI 형태만 먼저 잡고, 연동은 별도 단계에서 진행한다.
- 한 화면에서만 쓰는 UI는 성급하게 컴포넌트로 분리하지 않는다. 재사용이 생기거나 파일이 읽기 어려워질 때 분리한다.
- UI/CSS 변경이 브라우저에서 그대로 보이면 소스만 보지 말고 실제 dev 서버가 내려주는 CSS/JS 번들을 확인한다.
- Next/Turbopack dev 서버가 수정된 CSS를 반영하지 않으면 서버 재시작을 먼저 시도하고, 그래도 stale하면 `apps/frontend/.next` 캐시 삭제 후 재실행한다.
- 차트/canvas/시각화 UI는 DOM 존재 여부만 확인하지 말고 실제 브라우저 스크린샷 또는 렌더링된 픽셀/크기까지 확인한다.
- Tiptap 에디터 UI를 수정할 때는 전역 `.field`, `input`, `button`, `label` 스타일이 에디터 내부 DOM에 새는지 브라우저 computed style로 확인한다.
