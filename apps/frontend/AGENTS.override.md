# 프론트엔드 작업 지침

- 프론트엔드 코드는 Next.js App Router의 `app/`를 라우팅, 레이아웃, 페이지 조립 중심으로 둔다.
- 새 화면을 추가하거나 기존 페이지 파일이 크게 커질 때는 가능하면 `features/{feature}/` 아래로 실제 기능 코드를 옮긴다.
- 기능 폴더는 역할이 실제로 생겼을 때만 만든다. 구조를 맞추기 위해 비어 있거나 얇은 전달만 하는 `api`, `type`, `helper`, `component` 파일을 만들지 않는다.
- 백엔드의 `controller`, `service`, `helper`, `type` 구조를 프론트에 1:1로 복제하지 않는다.
- `app/{route}/page.tsx`는 해당 라우트의 데이터 연결, 상태 조립, 화면 조립을 담당하되, 파일이 커지면 렌더링 덩어리와 순수 로직부터 feature로 분리한다.
- `features/{feature}/`에는 해당 기능에서만 쓰는 화면 조각, API 호출, 타입, 순수 로직을 둔다.
- `components/ui/`에는 버튼, 인풋, 다이얼로그처럼 도메인 의미가 없는 공용 UI primitive만 둔다.
- `components/`에는 `TopNav`, `RichTextEditor`처럼 여러 기능에서 실제로 공유하는 UI만 둔다.
- `lib/`에는 feature 의미가 없는 공통 기술 코드만 둔다. 특정 기능에서만 쓰는 fetch 함수, 포맷터, 검증 함수는 해당 feature 안에 둔다.
- 여러 화면에서 재사용되지 않는 단순 타입은 별도 파일로 분리하지 말고 사용하는 파일 근처에 둔다.
- 같은 API 요청, 에러 처리, 타입 변환이 여러 화면에서 반복될 때만 공용 `api` 또는 `request` 유틸을 만든다.
- 대시보드 첫 화면은 루트 `AGENTS.md` 지침대로 데스크톱에서 핵심 섹션이 한눈에 보이는 구성을 우선한다.
- UI, 스타일, 레이아웃, 색상, 타이포그래피를 변경할 때는 루트 `DESIGN.md`를 기준으로 삼고, 의도적으로 디자인 기준을 바꾸는 작업이면 `DESIGN.md`도 함께 갱신한다.
- 사용자가 명시적으로 요청하기 전까지 프론트 구조 정리 중에 API 연동, 인증, 전역 상태관리, 라우팅 확장을 함께 끼워 넣지 않는다.

# 검증

- 프론트엔드 수정 후 가능한 경우 `pnpm --dir apps/frontend typecheck`, `pnpm --dir apps/frontend build` 중 변경 범위에 맞는 명령을 실행한다.
- UI/CSS 변경이 있으면 개발 서버와 브라우저에서 실제 화면을 확인한다.
- 문서만 수정한 경우에는 테스트 실행이 필수는 아니다.
